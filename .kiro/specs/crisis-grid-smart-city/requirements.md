# Requirements Document

## Introduction

This document defines requirements for extending the existing Crisis Grid Power Monitoring Dashboard into a **Multi-Infrastructure Smart City Monitoring Platform** for Bangalore, India. The existing Power Infrastructure Dashboard (BESCOM substations, live sensor simulation, cascade simulation, AI decision support) remains fully intact. The extension adds Water, Gas/LPG, Communications, and Physical Infrastructure monitoring across seven Bangalore zones: Whitefield, Yelahanka, Indiranagar, Hebbal, KR Puram, Electronic City, and Peenya.

The platform adds:
- A Map Layer Toggle System on the existing Bangalore map
- Dedicated Infrastructure pages (Water, Gas, Communications, Physical)
- A cross-infrastructure Failure Prediction module
- An enhanced AI Decision Support page backed by Amazon Bedrock
- AWS DynamoDB persistence for assets, events, sensor readings, and prediction results

## Glossary

- **Platform**: The Crisis Grid Multi-Infrastructure Smart City Monitoring Platform (the full React + Vite application)
- **Dashboard**: The existing top-level overview page (`Dashboard.jsx`)
- **BangaloreMap**: The existing Leaflet-based map component (`BangaloreMap.jsx`)
- **Layer_Toggle**: The UI control that shows or hides infrastructure-specific node overlays on the BangaloreMap
- **Infrastructure_Type**: One of: Power, Water, Gas, Communications, Physical
- **Asset**: A monitored infrastructure node (substation, pump station, pipeline segment, cell tower, bridge, etc.)
- **Asset_List**: A filterable, sortable table of Assets for a given Infrastructure_Type
- **Asset_Detail_Panel**: A slide-in or inline panel showing full sensor readings and metadata for a selected Asset
- **Sensor_Reading**: A timestamped record of sensor values for an Asset
- **Failure_Prediction_Module**: The page that scores all Assets across all Infrastructure_Types with a risk level
- **Risk_Level**: One of: Low (score < 0.35), Medium (score 0.35–0.69), High (score ≥ 0.70)
- **Heuristic_Scorer**: The client-side scoring function that computes a Risk_Level from Sensor_Reading values
- **AI_Decision_Support**: The page that presents active events, AI analysis, impact overview, and an AI Action Plan
- **Bedrock_Client**: The existing `useBedrock` hook that calls Amazon Bedrock Claude via the Lambda proxy
- **DynamoDB_Client**: The AWS SDK DynamoDB DocumentClient used to persist and query platform data
- **Lambda_Proxy**: The existing AWS Lambda function that forwards requests to Amazon Bedrock and DynamoDB
- **Active_Event**: An anomaly or predicted failure that has not been resolved
- **Impact_Overview**: A summary of population affected, regions impacted, and systems involved for an Active_Event
- **AI_Action_Plan**: A prioritised list of recommended actions generated by Amazon Bedrock for an Active_Event
- **Sidebar**: The existing collapsible navigation component (`Sidebar.jsx`)
- **Infrastructure_Section**: The "Infrastructure" group in the Sidebar containing sub-pages for each Infrastructure_Type
- **Zone**: One of the seven Bangalore geographic areas: Whitefield, Yelahanka, Indiranagar, Hebbal, KR Puram, Electronic City, Peenya
- **Top_Metrics**: The row of StatCard components at the top of each Infrastructure page showing key KPIs
- **StatCard**: The existing reusable metric card component (`StatCard.jsx`)

## Requirements

### Requirement 1: Map Layer Toggle System

**User Story:** As an operator, I want to toggle infrastructure layers on the Bangalore map, so that I can focus on one or more infrastructure types without visual clutter.

#### Acceptance Criteria

1. THE Layer_Toggle SHALL render as a set of toggle buttons overlaid on the BangaloreMap, one button per Infrastructure_Type.
2. WHEN an operator activates a Layer_Toggle button, THE BangaloreMap SHALL display Asset nodes for that Infrastructure_Type as CircleMarker elements on the map.
3. WHEN an operator deactivates a Layer_Toggle button, THE BangaloreMap SHALL remove all CircleMarker elements belonging to that Infrastructure_Type from the map.
4. THE Layer_Toggle SHALL support simultaneous activation of multiple Infrastructure_Type layers.
5. THE BangaloreMap SHALL render each Infrastructure_Type layer with a visually distinct colour: Power (blue `#3A86FF`), Water (cyan `#00E5FF`), Gas (amber `#f59e0b`), Communications (purple `#a855f7`), Physical (green `#39FF14`).
6. WHEN an operator hovers over an Asset node on the map, THE BangaloreMap SHALL display a Tooltip containing the Asset name, Zone, Infrastructure_Type, and current Risk_Level.
7. THE Layer_Toggle SHALL preserve the existing Power layer behaviour, transmission lines, and critical facility markers without modification.
8. WHERE the Platform is rendered on a viewport narrower than 768 px, THE Layer_Toggle SHALL collapse into a compact icon-only row to avoid obscuring the map.

### Requirement 2: Water Infrastructure Page

**User Story:** As an operator, I want a dedicated Water Infrastructure page, so that I can monitor Bangalore's water distribution network in real time.

#### Acceptance Criteria

1. THE Infrastructure_Section SHALL include a "Water" navigation entry that renders the Water Infrastructure page.
2. THE Water Infrastructure page SHALL display a Top_Metrics row using StatCard components showing: total pump stations online, active leaks detected, average network pressure (bar), and average flow efficiency (%).
3. THE Water Infrastructure page SHALL embed the BangaloreMap with the Water layer activated by default.
4. THE Water Infrastructure page SHALL display an Asset_List of water Assets (pump stations, reservoirs, pipeline segments) covering the seven Zones.
5. WHEN an operator selects an Asset from the Asset_List, THE Asset_Detail_Panel SHALL display: asset name, Zone, asset type, current pressure (bar), flow rate (L/s), leak detection status, pump health score (0–100), and last updated timestamp.
6. IF a water Asset's pressure drops below 1.5 bar, THEN THE Water Infrastructure page SHALL highlight that Asset in the Asset_List with a red indicator and increment the "active leaks detected" StatCard.
7. IF a water Asset's pump health score falls below 40, THEN THE Water Infrastructure page SHALL display a warning badge on that Asset in the Asset_List.
8. THE Water Infrastructure page SHALL simulate Sensor_Reading updates for all water Assets every 15 seconds using a client-side data generator.

### Requirement 3: Gas / LPG Supply Infrastructure Page

**User Story:** As an operator, I want a dedicated Gas Infrastructure page, so that I can monitor Bangalore's gas distribution network and detect pressure anomalies or leaks.

#### Acceptance Criteria

1. THE Infrastructure_Section SHALL include a "Gas" navigation entry that renders the Gas Infrastructure page.
2. THE Gas Infrastructure page SHALL display a Top_Metrics row using StatCard components showing: total pipeline segments monitored, active pressure anomalies, average distribution pressure (bar), and leak alerts count.
3. THE Gas Infrastructure page SHALL embed the BangaloreMap with the Gas layer activated by default.
4. THE Gas Infrastructure page SHALL display an Asset_List of gas Assets (pipeline segments, distribution stations, valve clusters) covering the seven Zones.
5. WHEN an operator selects an Asset from the Asset_List, THE Asset_Detail_Panel SHALL display: asset name, Zone, asset type, inlet pressure (bar), outlet pressure (bar), flow rate (m³/h), valve status (open/closed/partial), leak sensor reading (ppm), and last updated timestamp.
6. IF a gas Asset's leak sensor reading exceeds 50 ppm, THEN THE Gas Infrastructure page SHALL display a critical alert badge on that Asset and add an Active_Event to the AI_Decision_Support page.
7. IF a gas Asset's outlet pressure deviates more than 20% from its nominal value, THEN THE Gas Infrastructure page SHALL highlight that Asset with an amber warning indicator.
8. THE Gas Infrastructure page SHALL simulate Sensor_Reading updates for all gas Assets every 15 seconds using a client-side data generator.

### Requirement 4: Communications Infrastructure Page

**User Story:** As an operator, I want a dedicated Communications Infrastructure page, so that I can monitor cell towers, fibre nodes, and network uptime across Bangalore.

#### Acceptance Criteria

1. THE Infrastructure_Section SHALL include a "Communications" navigation entry that renders the Communications Infrastructure page.
2. THE Communications Infrastructure page SHALL display a Top_Metrics row using StatCard components showing: total communication nodes online, nodes with degraded signal, average uptime (%), and active outage count.
3. THE Communications Infrastructure page SHALL embed the BangaloreMap with the Communications layer activated by default.
4. THE Communications Infrastructure page SHALL display an Asset_List of communication Assets (cell towers, fibre nodes, exchange points) covering the seven Zones.
5. WHEN an operator selects an Asset from the Asset_List, THE Asset_Detail_Panel SHALL display: asset name, Zone, asset type, signal strength (dBm), uptime percentage, active connections count, backhaul status (online/degraded/offline), and last updated timestamp.
6. IF a communication Asset's uptime falls below 95%, THEN THE Communications Infrastructure page SHALL highlight that Asset with an amber warning indicator.
7. IF a communication Asset's backhaul status is "offline", THEN THE Communications Infrastructure page SHALL display a critical alert badge on that Asset.
8. THE Communications Infrastructure page SHALL simulate Sensor_Reading updates for all communication Assets every 15 seconds using a client-side data generator.

### Requirement 5: Physical Infrastructure Page

**User Story:** As an operator, I want a dedicated Physical Infrastructure page, so that I can monitor the structural health of bridges, buildings, and roads equipped with IoT sensors across Bangalore.

#### Acceptance Criteria

1. THE Infrastructure_Section SHALL include a "Physical" navigation entry that renders the Physical Infrastructure page.
2. THE Physical Infrastructure page SHALL display a Top_Metrics row using StatCard components showing: total physical Assets monitored, Assets with structural warnings, average structural health score (0–100), and critical alerts count.
3. THE Physical Infrastructure page SHALL embed the BangaloreMap with the Physical layer activated by default.
4. THE Physical Infrastructure page SHALL display an Asset_List of physical Assets (bridges, buildings, road segments) covering the seven Zones.
5. WHEN an operator selects an Asset from the Asset_List, THE Asset_Detail_Panel SHALL display: asset name, Zone, asset type (bridge/building/road), structural health score (0–100), vibration reading (mm/s), tilt angle (degrees), last inspection date, and last updated timestamp.
6. IF a physical Asset's structural health score falls below 50, THEN THE Physical Infrastructure page SHALL highlight that Asset with an amber warning indicator.
7. IF a physical Asset's structural health score falls below 25, THEN THE Physical Infrastructure page SHALL display a critical alert badge on that Asset and add an Active_Event to the AI_Decision_Support page.
8. THE Physical Infrastructure page SHALL simulate Sensor_Reading updates for all physical Assets every 30 seconds using a client-side data generator.

### Requirement 6: Infrastructure Section Navigation

**User Story:** As an operator, I want a grouped "Infrastructure" section in the sidebar, so that I can navigate between all infrastructure type pages without cluttering the top-level navigation.

#### Acceptance Criteria

1. THE Sidebar SHALL contain an "Infrastructure" section group that expands to reveal navigation entries for: Power (existing), Water, Gas, Communications, and Physical.
2. WHEN an operator clicks the "Infrastructure" section header, THE Sidebar SHALL toggle the visibility of the sub-navigation entries.
3. THE Sidebar SHALL display the Infrastructure section as expanded by default.
4. WHEN an operator navigates to any Infrastructure sub-page, THE Sidebar SHALL highlight that sub-page entry as active using the Infrastructure_Type colour.
5. THE Sidebar SHALL preserve all existing top-level navigation entries (Dashboard, Failure Prediction, Cascade Simulation, AI Decision Support, Historical Analytics, Incident Reports, Smart City) without modification.

### Requirement 7: Cross-Infrastructure Failure Prediction Module

**User Story:** As an operator, I want a unified Failure Prediction page that covers all infrastructure types, so that I can identify at-risk assets across the entire city before failures occur.

#### Acceptance Criteria

1. THE Failure_Prediction_Module SHALL display a system filter control allowing the operator to select one or more Infrastructure_Types to include in the risk view.
2. WHEN the operator selects an Infrastructure_Type filter, THE Failure_Prediction_Module SHALL update the Asset risk list to show only Assets of the selected Infrastructure_Type(s).
3. THE Failure_Prediction_Module SHALL compute a Risk_Level for each Asset using the Heuristic_Scorer based on current Sensor_Reading values.
4. THE Heuristic_Scorer SHALL assign Risk_Level "High" to an Asset when its computed risk score is greater than or equal to 0.70.
5. THE Heuristic_Scorer SHALL assign Risk_Level "Medium" to an Asset when its computed risk score is greater than or equal to 0.35 and less than 0.70.
6. THE Heuristic_Scorer SHALL assign Risk_Level "Low" to an Asset when its computed risk score is less than 0.35.
7. THE Failure_Prediction_Module SHALL render the BangaloreMap with all active Infrastructure_Type layers visible, where each Asset node is coloured according to its Risk_Level: Low (green `#39FF14`), Medium (amber `#FFB020`), High (red `#FF3B3B`).
8. THE Failure_Prediction_Module SHALL display a ranked Asset list sorted by descending risk score, showing asset name, Infrastructure_Type, Zone, risk score (%), and Risk_Level badge.
9. THE existing Power substation risk gauges and future risk predictions SHALL remain visible on the Failure_Prediction_Module when the Power filter is active.
10. WHEN the Heuristic_Scorer produces a risk score for a Power Asset, THE score SHALL be derived from the same failure_probability value already computed by `analyzeReading` in `mockData.js`.

### Requirement 8: Enhanced AI Decision Support Page

**User Story:** As an operator, I want an enhanced AI Decision Support page that covers all infrastructure types, so that I can get AI-generated analysis and action plans for any active event across the city.

#### Acceptance Criteria

1. THE AI_Decision_Support page SHALL display an Active_Events Panel listing all current Active_Events across all Infrastructure_Types, sorted by severity descending.
2. WHEN an operator selects an Active_Event from the Active_Events Panel, THE AI_Decision_Support page SHALL display an AI Analysis Panel with the Bedrock_Client analysis for that event.
3. THE AI Analysis Panel SHALL display: event summary, technical analysis text, cascade risk assessment, and estimated restoration time.
4. THE AI_Decision_Support page SHALL display an Impact_Overview section showing: estimated population affected (numeric), impacted Zones (list), and impacted Infrastructure_Types (list) for the selected Active_Event.
5. THE AI_Decision_Support page SHALL display an AI_Action_Plan section containing a prioritised list of recommended actions returned by the Bedrock_Client for the selected Active_Event.
6. WHEN the Bedrock_Client is unavailable, THE AI_Decision_Support page SHALL display a locally-generated fallback action plan derived from the Active_Event's Infrastructure_Type and severity.
7. THE AI_Decision_Support page SHALL preserve the existing Power-specific rerouting simulation, BESCOM protocol actions, and citizen notification features for Power Active_Events.
8. THE AI_Decision_Support page SHALL pass the Infrastructure_Type, Asset metadata, and current Sensor_Reading values to the Bedrock_Client as structured context when requesting analysis.
9. WHEN an Active_Event is resolved (severity returns to "low"), THE AI_Decision_Support page SHALL remove that event from the Active_Events Panel.

### Requirement 9: DynamoDB Data Persistence

**User Story:** As a platform administrator, I want infrastructure asset data, sensor readings, and prediction results stored in DynamoDB, so that the platform has a persistent data layer for historical queries and audit trails.

#### Acceptance Criteria

1. THE DynamoDB_Client SHALL write a Sensor_Reading record to the `crisis-grid-sensor-readings` DynamoDB table each time a Sensor_Reading is generated for any Asset.
2. THE DynamoDB_Client SHALL write an Asset record to the `crisis-grid-assets` DynamoDB table when a new Asset is registered in the Platform.
3. THE DynamoDB_Client SHALL write a prediction result record to the `crisis-grid-predictions` DynamoDB table each time the Heuristic_Scorer computes a Risk_Level for an Asset.
4. THE DynamoDB_Client SHALL write an Active_Event record to the `crisis-grid-events` DynamoDB table when an Active_Event is created.
5. WHEN a DynamoDB write operation fails, THE DynamoDB_Client SHALL log the error to the browser console and continue Platform operation without interrupting the UI.
6. THE `crisis-grid-sensor-readings` table SHALL use a composite primary key of `assetId` (partition key) and `timestamp` (sort key).
7. THE `crisis-grid-assets` table SHALL use `assetId` as the partition key.
8. THE `crisis-grid-predictions` table SHALL use a composite primary key of `assetId` (partition key) and `timestamp` (sort key).
9. THE `crisis-grid-events` table SHALL use `eventId` as the partition key.

### Requirement 10: Asset Data Model

**User Story:** As a developer, I want a consistent data model for all infrastructure assets, so that shared components (map, asset list, detail panel, prediction module) can render any asset type without type-specific branching.

#### Acceptance Criteria

1. THE Platform SHALL represent every Asset with a base schema containing: `assetId` (string), `name` (string), `shortName` (string), `infraType` (Infrastructure_Type), `zone` (Zone), `lat` (number), `lng` (number), `status` (string), and `lastUpdated` (ISO timestamp).
2. THE Platform SHALL extend the base Asset schema with infrastructure-specific sensor fields for each Infrastructure_Type without removing base fields.
3. THE Platform SHALL define Water Assets with additional fields: `pressureBar` (number), `flowLps` (number), `leakDetected` (boolean), `pumpHealthScore` (number 0–100).
4. THE Platform SHALL define Gas Assets with additional fields: `inletPressureBar` (number), `outletPressureBar` (number), `flowM3h` (number), `valveStatus` (string), `leakPpm` (number).
5. THE Platform SHALL define Communications Assets with additional fields: `signalDbm` (number), `uptimePct` (number), `activeConnections` (number), `backhaulStatus` (string).
6. THE Platform SHALL define Physical Assets with additional fields: `structuralHealthScore` (number 0–100), `vibrationMms` (number), `tiltDegrees` (number), `lastInspectionDate` (string).
7. THE Platform SHALL define Power Assets using the existing substation schema from `bangaloreData.js` without modification.
8. FOR ALL Asset records written to DynamoDB, the `assetId` field SHALL be unique across all Infrastructure_Types within the Platform.

### Requirement 11: Sensor Data Simulation

**User Story:** As a developer, I want client-side sensor data generators for each new infrastructure type, so that the Platform can operate with realistic simulated data before live IoT feeds are connected.

#### Acceptance Criteria

1. THE Platform SHALL provide a `generateWaterReading(assetId)` function that returns a Sensor_Reading with randomised but realistic values for all Water Asset sensor fields.
2. THE Platform SHALL provide a `generateGasReading(assetId)` function that returns a Sensor_Reading with randomised but realistic values for all Gas Asset sensor fields.
3. THE Platform SHALL provide a `generateCommsReading(assetId)` function that returns a Sensor_Reading with randomised but realistic values for all Communications Asset sensor fields.
4. THE Platform SHALL provide a `generatePhysicalReading(assetId)` function that returns a Sensor_Reading with randomised but realistic values for all Physical Asset sensor fields.
5. WHEN `generateWaterReading` is called with `forceAnomaly = true`, THE function SHALL return a Sensor_Reading where `pressureBar` is less than 1.5 or `pumpHealthScore` is less than 40.
6. WHEN `generateGasReading` is called with `forceAnomaly = true`, THE function SHALL return a Sensor_Reading where `leakPpm` is greater than 50 or outlet pressure deviation exceeds 20% of nominal.
7. WHEN `generateCommsReading` is called with `forceAnomaly = true`, THE function SHALL return a Sensor_Reading where `uptimePct` is less than 95 or `backhaulStatus` is "offline".
8. WHEN `generatePhysicalReading` is called with `forceAnomaly = true`, THE function SHALL return a Sensor_Reading where `structuralHealthScore` is less than 50.
9. THE Platform SHALL seed asset data for each Zone with a minimum of 3 Assets per Infrastructure_Type per Zone, totalling at least 21 Assets per new Infrastructure_Type.

### Requirement 12: Existing Power Infrastructure Preservation

**User Story:** As an operator, I want all existing Power monitoring features to continue working exactly as before, so that the extension does not regress any current functionality.

#### Acceptance Criteria

1. THE Platform SHALL preserve the existing Dashboard page, Monitoring page, FailurePrediction page, CascadeSimulation page, DecisionSupport page, HistoricalAnalytics page, Incidents page, and SmartCity page without functional changes.
2. THE Platform SHALL preserve the existing `bangaloreData.js`, `mockData.js`, `cascadeEngine.js`, and `incidentSolutions.js` data modules without modification.
3. THE Platform SHALL preserve the existing `BangaloreMap` component's Power layer rendering, transmission lines, critical facility markers, and reroute animation behaviour.
4. THE Platform SHALL preserve the existing `useBedrock` hook interface and Lambda proxy integration without modification.
5. WHEN the Platform loads, THE existing Power substation sensor simulation loop (10-second interval in `App.jsx`) SHALL continue to run unmodified.
6. THE Platform SHALL preserve the existing `AppContext` toast notification system without modification.
7. THE Platform SHALL preserve the existing design system CSS variables, card styles, badge styles, and animation classes defined in `index.css` without modification.
